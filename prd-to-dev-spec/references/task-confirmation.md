# Development Task Confirmation

Use this reference to create the pre-development confirmation sheet.

## Purpose

The confirmation sheet is a gate before coding. It verifies that product scope, technical interpretation, test coverage, dependencies, and owners are clear enough for programmers to start.

## Confirmation Categories

| Category | Questions |
|----------|-----------|
| Product baseline | Is PRD approved and versioned? |
| Scope | Are V1, deferred, and out-of-scope items clear? |
| Function detail | Are menus, buttons, business logic, and data logic clear? |
| API/data | Are interfaces, fields, validation, and migration clear? |
| Database scripts | Are SQL/migration/rollback/check script drafts generated by type and table, marked not executed, and assigned to DBA/data engineer review? |
| UI/prototype | Is sketch/prototype sufficient for development? |
| Test | Do test cases cover all AC and high-risk logic? |
| Dependencies | Are external systems, accounts, data, and configs ready? |
| Risks | Are unresolved issues assigned owners and dates? |

## Task Splitting Rules

Split tasks by:

1. Independent deliverability.
2. Clear owner and review method.
3. Natural module/API/data boundary.
4. Testable completion standard.

Avoid tasks like "finish backend" or "optimize page". Prefer:

```text
DEV-001 Implement bill creation API with validation and audit log.
DEV-002 Implement bill list page filters and pagination.
DEV-003 Add P0 test cases for amount validation and permission denial.
```

## Blocker Handling

If a blocker remains, do not hide it. Use:

```markdown
| Blocker | Impact | Owner | Required decision | Due date |
|---------|--------|-------|-------------------|----------|
```
