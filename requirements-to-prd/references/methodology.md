# 需求到 PRD 方法论地图（参考全文）

> 本文件为 [../SKILL.md](../SKILL.md) 的理论与术语参考；撰写 PRD 时以 SKILL 中的**产出模板与执行顺序**为准。

## 总览

产品经理从需求走到 PRD，本质是把模糊问题压缩成可被设计、研发、测试、业务共同执行的方案。新版技能默认拆成两份交付物：**需求分析文档**承载问题、根因、功能原子、方案可行性与范围建议；**PRD**承载确定范围后的功能、规则、数据、验收与交付口径。全过程仍可拆成 7 层：**机会识别、问题定义、需求洞察、方案设计、需求表达、优先级规划、验证交付**。EARS 属于「需求表达层」工具之一。

**简化链路**：

`战略目标` → `发现机会` → `定义问题` → `洞察用户与场景` → `设计方案` → `规范表达需求` → `做优先级与版本决策` → `评审、开发、测试、上线` → `数据验证与复盘`

**方法到层的映射**：

| 层 | 常用方法 |
|----|----------|
| 机会识别 | OKR / KPI、漏斗、市场与竞品、北极星 |
| 问题定义 | JTBD、5 Whys、Problem Statement、HMW |
| 需求洞察 | 访谈、Persona、Journey、服务蓝图、现状流程、数据 |
| 方案设计 | Design Thinking、Use Case、User Story、BPMN、状态机、信息架构、领域模型、MVP |
| 需求表达 | EARS、GWT、INVEST、IEEE/ISO 质量原则、NFR 框架 |
| 优先级 | MoSCoW、RICE、Kano、价值-成本、Roadmap |
| 验证闭环 | 原型/可用性测试、可研、需求评审、A/B、上线指标、追溯矩阵 |

## PRD 的四个层次（概念模型）

> 原文在「方案层」处截断；第四层为基于常见产品实践的**补全**，与七层法中的「需求表达 + 优先级 + 验证」对应。

- **目标层**：为什么做、解决什么业务问题、影响什么指标。  
- **用户层**：谁用、什么场景、痛点与目标。  
- **方案层**：范围、功能模块、流程、关键对象与状态、异常与边界、MVP。  
- **需求与验收层**（补全名）：用 PRD 固定写法表达功能/规则/非功能/权限/数据；用 EARS、GWT 等可验证；并包含优先级、版本边界与上线后验证指标。

## 各层要点（精简）

### 1 机会识别

**问题**：为什么现在做。方法：战略拆解、OKR/KPI、北极星、AARRR、市场/竞品。  
**产出**（常不进 PRD 正文，可进「背景/立项依据」附件）：机会点、业务目标、指标证据、是否立项的初步判断。

### 2 问题定义

**问题**：把「要加功能」改写成「要解决的问题」。方法：JTBD、5 Whys、Problem Statement、HMW。  
**产出**：目标用户、场景、根因、对业务与用户的影响。

### 3 需求洞察

**问题**：问题发生的真实环境。方法：访谈、Persona、Journey、服务蓝图、现状流程、数据分析。  
**产出**：用户分类、核心场景、当前流程、痛点、规则与系统约束。

### 4 方案设计

**问题**：做什么、怎么闭环。方法：Design Thinking、Use Case、User Story、BPMN、状态机、信息架构、领域建模、MVP。  
**产出**：方案范围、业务流程、功能模块、关键对象与状态、异常与边界、MVP 范围。  
**配图**：何种需求应附流程/时序/架构等图，见 [diagram-guide.md](diagram-guide.md)。

### 5 需求表达

**问题**：多角色理解一致。方法：EARS、GWT、INVEST、需求质量原则、NFR。  
**PRD 常见结构**：背景与目标、用户与场景、范围、业务流程、功能需求、业务规则、异常、权限、数据定义、NFR、验收、埋点与指标。  
**组合关系**：User Story 表达为什么；Use Case 展开业务怎么发生；EARS 规范系统必须做什么；GWT 定义怎么算完成。

### 6 优先级与版本

**问题**：先做什么、不做什么。方法：MoSCoW、RICE、Kano、价值-成本、Roadmap。  
**产出**：V1 必做、后续版本、明确排除项、依赖与排期说明。

### 7 验证与闭环

**问题**：PRD 可执行且可验证。方法：原型/可用性、技术评审、需求评审、A/B、上线指标、需求追溯矩阵。  
**产出**：评审结论、风险、测试口径、上线观测指标、复盘结论（可进「附录/评审记录」）。
