# PRD 配图规则（流程 / 时序 / 架构等）

> 本文件为 [../SKILL.md](../SKILL.md) 的**选配指南**：不增加 PRD 必选章节，但约定**在什么需求特征下**应生成**哪类图**，避免「每份 PRD 堆图」或「该澄清处无图」。

## 1. 总原则（推断链）

1. **一图一事**：每张图只回答一个问题（例如「主流程怎么走」与「谁调谁」不要挤在一张里，除非极简）。  
2. **图服务于歧义消除**：若文字步骤已唯一、无分支、无多角色，**不必**为配图而配图。  
3. **优先 Mermaid**：与 SKILL 模板 §4.2 一致，使用 `flowchart` / `sequenceDiagram` / `stateDiagram` 等，便于版本库与飞书 Markdown 渲染（具体语法以渲染器为准）。  
4. **图前必有导语**：用 1～2 句话说明「读图目的」与「未画进去的范围」（如「不含对账子流程」）。  
5. **与事实一致**：涉及外部系统名、接口、部署拓扑时，无依据则标注 **`待补充`**，不凭想象补全架构细节。

## 2. 需求信号 → 推荐图类型

用下表的**左侧信号**匹配当前需求；命中越多，越建议生成对应图（可同时多种）。

| 需求侧信号（命中即考虑） | 推荐图类型 | Mermaid 常用类型 | 建议落位（PRD 章节） |
|--------------------------|------------|------------------|----------------------|
| 多步操作、分支、循环、审批环、「视条件而定」 | **业务流程图** | `flowchart` | §4.2 主流程；复杂时 §3.3 现状流程也可 |
| 两个及以上**角色/系统**按时间先后交互、需对齐接口边界 | **时序图** | `sequenceDiagram` | §4.2 或紧挨 UC 表；跨系统写在 §8.3 附近亦可 |
| 同一对象有多种状态、状态迁移触发不同行为 | **状态图** | `stateDiagram-v2` | §4.2「关键对象与状态」；与 FR 中 State-driven 呼应 |
| 多个模块/子系统/第三方依赖，需讲清**边界与依赖方向** | **架构 / 容器级示意图** | `flowchart` 分层或 `graph` 简图（非强制 C4） | §4.1 或 §8.3；仅到「容器/子系统」粒度，避免过早画类图 |
| 数据从哪来、经哪些步骤、落到哪张表/主题 | **数据流图（轻量）** | `flowchart`（数据泳道可选） | §8.2 前后或 §4.2；细节不足则文字 + `待补充` |
| 多入口导航、深层信息架构、角色可见菜单差异大 | **信息架构 / 导航树** | `flowchart` 树状 或 `mindmap`（若渲染支持） | §4.1 |
| 实体多、关系易混、验收依赖字段一致性 | **ER / 领域关系草图（轻量）** | `erDiagram` | §8.2；字段级以表为准，图只表达关系 |
| 目标与手段层级多、需对齐战略到功能（可选） | **目标分解树** | `mindmap` 或 `flowchart` | §1 附录或独立「附图」小节；**勿替代** §11 优先级表 |

## 3. 什么情况下「应」配图（硬性建议）

满足**任一**即建议在 PRD 中至少增加**一张**对应类型图（或等价的分步表 + 状态表，但流程/时序类仍以图为佳）：

- **A. 强流程**：≥3 个业务步骤，且存在分支、回退、或「需人工审批/例外处理」。→ **流程图**。  
- **B. 强协作**：≥2 个角色或系统，且存在请求/响应、异步回调、超时重试。→ **时序图**（可多张：主成功路径 + 关键异常）。  
- **C. 强状态**：业务对象（订单、工单、账单、任务等）≥3 个状态，或状态决定可见操作。→ **状态图**。  
- **D. 强集成**：明确出现外部系统、开放平台、消息队列、定时批处理。→ **架构 / 容器示意图** + 必要时 **时序图**。  
- **E. 强数据**：多源汇聚、转换、落库，且测试验收依赖「数据口径」。→ **轻量数据流图** + §8.2 表。

## 4. 什么情况下「不必」配图（默认文字即可）

- 单表单、单接口、线性 CRUD，且无状态机、无跨系统。  
- 需求仅为规则列表、展示文案、单指标报表，无「谁先谁后」争议。  
- 信息已用 **GWT 表 + FR** 唯一确定行为，评审无人提出流程歧义（可事后补图）。

## 5. 图类型与内容边界（避免错误）

| 图类型 | 适合画什么 | 避免画什么 |
|--------|------------|------------|
| 流程图 | 业务决策与步骤、异常分支 | 把每个 UI 点击都画成节点（除非 UX 评审需要） |
| 时序图 | 系统间消息、同步/异步 | 把内部每个函数调用都展开（除非性能/一致性评审需要） |
| 状态图 | 对象生命周期、合法迁移 | 把页面路由与业务状态混为一谈（应分开或标注） |
| 架构图 | 子系统、依赖、信任边界 | 未确认的部署 IP、未选型的中间件细节 |
| ER 图 | 实体关系、基数 | 与 §8.2 表格冲突的字段臆造 |

## 6. 与七层法的对应关系（便于写稿时自检）

| 层 | 常见配图 |
|----|----------|
| 机会 / 问题 | 通常不画图；战略拆解可选目标树 |
| 洞察 | 用户旅程（简化 flowchart / 表格）；现状流程 |
| 方案 | **流程图、状态图、架构图** 主战场 |
| 需求表达 | 复杂 UC 可补 **时序图**；与 FR 编号交叉引用 |
| 数据 / 集成 | **ER / 数据流**；接口多时 **时序** |
| 验证 | 一般不新画图；可用简单流程表达埋点触发 |

## 7. Agent 执行时的最小规则（可嵌入实现）

1. 扫描需求正文：若出现「审批」「状态」「回调」「对接」「同步」「导入导出」「多角色」等词，按 §3 触发检查。  
2. 若生成流程图，**至少**包含：起点、终点、一个异常/拒绝分支（若适用）。  
3. 若生成时序图，**至少**标明：调用方、接收方、一条主成功消息链；可选虚线标注异步。  
4. 所有图在 PRD 中**引用 § 小节标题**，并在 FR/AC 中用「见图 x」或「见 §4.2」建立追溯。  
5. 无法确认事实时：图中用 `待补充` 节点或虚线框，与文字 **`待补充`** 一致。

## 8. 与 `demo/expected-prd.md` 的示例关系

个人账单类需求通常命中 **强流程**（录入→分类→报表）与 **强状态**（草稿/已确认），示例 PRD 中已含 **状态图**；若补充「对接银行」「多设备同步」，则应追加 **时序图** 与 **架构草图**。示例仅作结构参考，配图数量以真实需求信号为准。
